Three problems I'm glad existed

Diagnosis is the job. These are the three I'd tell you about in an interview.

// 01 — the batch that ate payroll day

Loading Dock's twice-weekly production batch was taking 14–16 hours to import a week of visit history — agencies waited nearly a full day on payroll day.

I traced it to redundant inter-service lookups — the same Employee and Consumer records re-fetched on every visit, dozens of times per pair per import. Fix: a job-scoped Redis cache keyed on the import file, cleared on completion.

The judgment call: I chose deduplication over throwing workers at it deliberately — parallel processing risked duplicate timesheets with downstream payroll impact. Correctness constrained the solution space.

before · per batch run 14–16 hrs
after · per batch run 2–3 hrs
≈ 80% reduction · direct client appreciation
// 02 — the token time bomb

During our GitHub → GitHub Enterprise migration, I spotted a silent failure during a routine deploy: one shared access token, used by every microservice to fetch client configuration, had been invalidated.

The trap: running servers looked perfectly healthy — they were serving cached config. Any restart would re-fetch with the dead token and take the site down. Staging and production sites began failing one by one across tenants.

There was no shortcut — every microservice needed redeployment with the new token. I took ownership with my engineering manager, coordinated fixes across every team and server, and supported the rollout in real time. All tenants stable within one day. Early identification was the difference between a one-day fix and an outage nobody could have recovered quickly.

// 03 — the every-other-monday crash

Every alternate Monday — payroll day — the platform crashed under biweekly visit-export load. Predictable, recurring, business-critical.

I led a layered investigation that resolved five stacked bottlenecks, each one exposing the next: staging and production sharing node pools; every tenant's export job firing at the same instant (staggered them — smoothed load and cut cost); pods too under-threaded to ever cross the HPA's 80% CPU threshold, so autoscaling silently never fired; DB connections exhausted (recalculated as max pods × workers × threads); and residual slowness from an upstream VM-hosted service with a too-narrow Passenger pool.

No recurrence since. Resource utilization healthy, exports fast, payroll Mondays boring.
